# EpiCase — Руководство администратора платформы (ADMIN_GUIDE.md)

Аудитория: пользователь с ролью **admin** внутри EpiCase (не IT-отдел!).
Задачи admin'а: управление учётками, группами, бэкапами, мониторинг, системные настройки.

Соответствует §14 PROJECT_DESIGN_EPICASE_v1, ADDENDUM §SCALE.

---

## 1. Вход

1. Открыть в браузере `http://epicase.vmeda.local/`.
2. Логин: `admin`. Пароль: значение `FIRST_ADMIN_PASSWORD` из `.env`, выданное IT-отделом при развёртывании (см. `docs/DEPLOY.md` §5).
3. При первом входе система потребует сменить пароль. Новый пароль должен быть не короче 12 символов и содержать буквы + цифры.
4. После входа открывается AdminDashboard. В правом верхнем углу — HealthWidget (см. §7).

Если пароль забыт — IT-отдел должен пересоздать первого admin'а через `docker compose exec server python -m scripts.seed`.

## 2. Создание учителей и студентов

Раздел: **Admin → Пользователи**.

### Создание по одному (single)

1. Нажать кнопку «**Создать пользователя**» (правый верхний угол).
2. В модалке «Создать пользователя» заполнить:
   - **Логин** — латиница, без пробелов (например, `ivanov.i`).
   - **Пароль** — временный, пользователь сменит при первом входе.
   - **ФИО** — полное имя как в зачётке.
   - **Роль пользователя** — Студент / Преподаватель / Администратор.
   - **Группа пользователя** — для студентов выбрать группу, иначе «Без группы».
3. Нажать «**Создать**». В таблице появится новая строка.

### Bulk-импорт через CSV

1. На странице «Пользователи» нажать «**Импорт из CSV**».
2. В модалке нажать «**Скачать шаблон CSV**» — сохранится `users_template.csv`.
3. Открыть шаблон в LibreOffice / Excel (разделитель `;`, кодировка UTF-8) и заполнить.

**Формат CSV:**
```
username;password;full_name;role;group_name;email
ivanov.i;Student123!;Иванов И.И.;student;431 учебная;
petrov.p;Student123!;Петров П.П.;student;431 учебная;
sidorov.s;Teacher123!;Сидоров С.С.;teacher;;sidorov@vmeda.mil
```

Поля:
- `role` — одно из `student`, `teacher`, `admin`.
- `group_name` — должна быть создана заранее (см. §3); для teacher/admin оставлять пустым.
- `email` — опционально.

4. Загрузить файл через input «CSV файл». Сразу показывается **preview первых 10 строк**.
5. Нажать «**Загрузить CSV**».
6. Поведение all-or-nothing: при любой ошибке (422) ни одна строка не сохранится. Список ошибок выводится в красном блоке с указанием номера строки и причины (`Строка 7: дубликат username`).
7. После исправления CSV — повторить загрузку.

### Действия над пользователем

В таблице каждая строка имеет кнопки:
- **Edit** — изменить ФИО / группу.
- **Reset password** — задать новый временный пароль (пользователь обязан сменить при следующем входе).
- **Block / Unblock** — заблокировать (`is_active=false`) или разблокировать.
- **Delete** — soft-delete (фактически = блокировка). Полное удаление недоступно, чтобы сохранить связи с результатами.

### Фильтры

Над таблицей: поиск по логину/ФИО, фильтр по роли, статусу, группе.

## 3. Управление группами

Раздел: **Admin → Группы** (либо API `/api/groups`).

### Создание группы

1. Нажать «**Создать группу**».
2. Поле «Название» — например, `431 учебная`. Должно быть уникальным.
3. «Создать» → группа появляется в списке.

### Назначение преподавателя группе

1. Выбрать группу в таблице → кнопка «**Назначить преподавателя**».
2. В модалке выбрать teacher'а из выпадающего списка.
3. Один преподаватель может вести несколько групп; одна группа может иметь несколько преподавателей (M2M `teacher_groups`).
4. Снятие — кнопка «Удалить» рядом с именем преподавателя.

### Добавление студентов в группу

Два способа:
- При создании студента указать `group_id` (см. §2).
- В таблице студент: «Edit» → выбрать группу из списка.

Удаление студента из группы — «Edit» → «Без группы».

## 4. Бэкапы

Раздел: **Admin → Система** → секция «Бэкапы».

### Ручное создание

Нажать кнопку «**Создать бэкап**» (правый верхний угол секции). Спиннер на кнопке → через ~5–60 с (зависит от размера БД) в таблице появится новая запись:
- **Имя файла** — `epicase_YYYYMMDD_HHMMSS.sql`.
- **Размер** — в МБ.
- **Создан** — дата/время.
- **Возраст** — human-readable («3 минуты назад»).

### Автоматическое расписание

Cron внутри контейнера server запускает бэкап **ежедневно в 02:00 UTC** (06:00 МСК). Включать/выключать через UI нельзя — это часть baseline-инфраструктуры.

### Retention

Срок хранения настраивается в **Admin → Настройки** → «Backup retention, дней» (по умолчанию **90**, допустимо 7–365). Старые бэкапы удаляются автоматически.

### Список и скачивание

В таблице бэкапов каждая строка:
- «**Скачать**» — отдаёт файл (`/api/admin/backup/<filename>`).
- «**Восстановить <filename>**» — открывает triple-confirm flow (см. §5).
- «**Удалить**» — одиночный confirm → бэкап удаляется навсегда.

## 5. Restore — детально

Восстановление БД из бэкапа — деструктивная операция, поэтому защищена тремя подтверждениями.

### Шаг 1. Initial confirm

Нажать «**Восстановить <filename>**» → откроется ConfirmDialog:
> «Это заменит все данные текущей БД данными из бэкапа. Все активные попытки будут прерваны.»
> Кнопки: **Отмена** / **Я понимаю**.

### Шаг 2. Ввод имени бэкапа

Откроется Modal «Введите имя бэкапа»:
> «Введите имя бэкапа для подтверждения: **epicase_20260518_021500.sql**»
> Поле ввода. Кнопка «**Продолжить**» доступна **только** при точном совпадении введённого текста с именем файла.

### Шаг 3. Final confirm

Откроется ConfirmDialog «**ПОДТВЕРДИТЕ восстановление**»:
> «Финальное подтверждение. После запуска восстановление нельзя отменить, а текущая сессия будет завершена после окончания операции.»
> Кнопки: **Отмена** / **Восстановить систему**.

### Поведение после запуска

1. Сервер переводит систему в **maintenance_mode** (`POST /api/admin/restore/{filename}` → 202 Accepted).
2. UI поллит `/api/admin/sysinfo` каждые 5 секунд, отображает баннер «Система в режиме обслуживания».
3. Когда `maintenance_mode` возвращается в `false` — UI показывает toast «Восстановление завершено» → автоматический **logout** → редирект на `/login?reason=system_restored`.
4. Все активные сессии (включая admin'а, запустившего restore) аннулируются.
5. После повторного входа — новые данные из бэкапа.

### Что делать, если restore «завис»

- Подождать минимум 10 минут (большие БД).
- Если HealthWidget показывает `error` и сессия не разлогинилась — обратиться в IT (см. `docs/DEPLOY.md` §10).

## 6. Просмотр логов

Раздел: **Admin → Система** → секция «Логи системы».

- Фильтр **Уровень логов**: All / DEBUG / INFO / WARNING / ERROR / CRITICAL.
- Кнопка «**Экспорт CSV**» — выгружает текущую страницу в `system_logs.csv` (разделитель `;`, поля: `id;level;message;username;created_at`).
- Под таблицей — пагинация: «Страница N из M», 20 записей на страницу.
- Цвет бейджа: ERROR/CRITICAL — красный, WARNING — жёлтый, остальные — нейтральный.

Логи также пишутся в stdout контейнера server (для IT — `docker compose logs server`).

## 7. HealthWidget

Виджет в правом верхнем углу AdminDashboard. Опрашивает `/api/health` каждые 30 секунд.

| Цвет / статус | Значение                                                                       | Действие admin'а                                         |
|---------------|--------------------------------------------------------------------------------|----------------------------------------------------------|
| **ok** (зелёный)    | БД, диск, бэкапы — всё в норме.                                          | Ничего.                                                  |
| **warning** (жёлтый) | Например: диск <20% свободно, или последний автобэкап старше 48 ч.       | Удалить старые бэкапы / уведомить IT.                    |
| **error** (красный)  | БД недоступна, или диск <5%, или критическая ошибка миграции.            | **Звуковой алерт** (1×). Немедленно открыть IT-инцидент. |

При переходе в `error` виджет однократно проигрывает короткий звуковой сигнал (browser AudioContext). Повторный звук — только после очередной смены `ok/warning → error`.

## 8. Системные настройки

Раздел: **Admin → Настройки**.

| Поле                       | Тип    | Диапазон  | По умолчанию | Назначение                                            |
|----------------------------|--------|-----------|--------------|-------------------------------------------------------|
| **Название учреждения**    | text   | до 200    | пусто        | Отображается в шапке и PDF-отчётах.                   |
| **Idle timeout, мин**      | number | 5–120     | 30           | Через сколько минут бездействия пользователь логаутится. |
| **Max file upload, МБ**    | number | 1–50      | 5            | Лимит на загрузку картинок в сценарии.                |
| **Backup retention, дней** | number | 7–365     | 90           | Срок хранения автоматических бэкапов.                 |

Алгоритм работы:
1. Изменить значения → появится жёлтый баннер «Есть несохранённые изменения».
2. Нажать «**Сохранить настройки**».
3. Toast «Настройки сохранены».
4. При попытке закрыть вкладку с несохранёнными изменениями браузер покажет `beforeunload` confirm.

## 9. Что НЕ делать

Ограничения, защищённые backend'ом (попытка вернёт 400/403):

- **Нельзя deactivate себя** — admin не может заблокировать собственную учётку (`PUT /api/users/{self.id}/status` с `is_active=false` → 400). Если нужно — попросить другого admin'а.
- **Нельзя удалить последнего admin'а** — система запрещает блокировать или менять роль единственного активного администратора. Сначала создать второго admin'а, затем менять первого.
- **Нельзя править свою же роль** — смена роли admin → teacher/student на собственной учётке заблокирована.
- **Нельзя восстанавливать бэкап от другой major-версии** — restore проверяет совместимость схемы; при несовместимости → 409 Conflict. Сначала apply миграций, потом restore.
- **Нельзя загружать CSV с дубликатами username** — all-or-nothing: ни одна строка не сохранится.
- **Не менять `.env` через UI** — секреты (JWT_SECRET, POSTGRES_PASSWORD) меняются только IT-отделом на сервере.

## 10. Troubleshooting

### Не приходит CSV-импорт — все строки отвалились
- Проверить кодировку (должна быть UTF-8, не Windows-1251).
- Проверить разделитель (`;`, а не `,`).
- Проверить, что `group_name` существует в системе (создать группы заранее).
- Прочитать список ошибок в красном блоке — каждая строка с указанием причины.

### Кнопка «Восстановить» неактивна
Имя в поле ввода не совпадает с именем файла (regex-чувствительно к пробелам и регистру). Скопировать имя из таблицы.

### HealthWidget постоянно warning
Открыть `/api/health` напрямую в браузере, посмотреть какой именно check провален (`db`, `disk`, `backups`). Если `disk` — удалить старые бэкапы или попросить IT расширить диск.

### Списки пользователей / групп не обновляются
TanStack Query кэширует данные. Hard refresh (`Ctrl+F5`) или подождать 30 с (auto-refetch).

### Восстановление длится >15 минут
Скорее всего БД очень большая или диск медленный. Не закрывать вкладку (поллинг продолжается), не делать рестарт контейнеров. Если по логам сервера всё «висит» — IT.

### Забыл пароль admin'а
Только IT может: `docker compose exec server python -m scripts.reset-admin-password` (см. `docs/DEPLOY.md` §5).
