# Руководство преподавателя EpiCase v1.0.0

Гайд для преподавателей кафедр ВМедА: создание клинических сценариев,
проверка, назначение группе и анализ результатов студентов.

Источник истины по типам узлов и логике: `docs/PROJECT_DESIGN_EPICASE_v1.md`
+ `docs/PROJECT_DESIGN_ADDENDUM_v1.1.md`.

---

## 1. Вход преподавателя

1. Откройте `https://epicase.vmeda.local` (или адрес, выданный администратором).
2. Логин и временный пароль выдаёт администратор кафедры.
3. После первого входа система потребует **сменить пароль**.
4. После авторизации вы попадёте на страницу `Teacher dashboard`.
   В левом меню видны разделы: «Мои сценарии», «Группы», «Аналитика».

> Если кнопка «Войти» не реагирует — проверьте, что нет блокирующего
> расширения браузера, и обновите страницу (Ctrl+F5).

---

## 2. Создание нового сценария

1. Перейдите в **«Мои сценарии»** (`/teacher/scenarios`).
2. Нажмите **«Create scenario»** в правом верхнем углу.
3. Заполните поля:
   - **Название** (`title`) — короткое и информативное. Пример: *«Острый
     гастроэнтерит у курсанта, март 2026»*.
   - **Описание** (`description`) — 1–3 предложения о клинической ситуации
     и учебных целях.
   - **Лимит времени** (`time_limit_sec`) — общая длительность кейса
     в секундах. Стандартно для эпидкейсов — 1800 (30 мин) или 2700 (45 мин).
4. После создания сценарий получает статус **`draft`** (черновик) и
   открывается в редакторе графа.

---

## 3. Граф-редактор — обзор

Редактор состоит из трёх колонок:

| Колонка       | Назначение                                                 |
|---------------|------------------------------------------------------------|
| Слева         | **Палитра узлов** (Node palette) — 6 типов, drag-and-drop. |
| По центру     | **Canvas** на @xyflow/react — здесь живёт граф.            |
| Справа        | **Inspector** — настройки выбранного узла.                 |

### 3.1. Шесть типов узлов

| Тип          | Иконка / цвет          | Что делает                                            |
|--------------|------------------------|-------------------------------------------------------|
| `start`      | зелёный                | Точка входа. В графе ровно один.                     |
| `data`       | голубой                | HTML-материал для прочтения (анамнез, лаборатория).  |
| `decision`   | синий (royal)          | Выбор варианта(ов) ответа.                            |
| `form`       | фиолетовый             | Заполнение полей формы (например, диагноз + КОД МКБ). |
| `text_input` | бирюзовый              | Свободный текстовый ответ с keyword-проверкой.       |
| `final`      | жёлтый                 | Финальный экран с порогом прохождения.                |

### 3.2. Drag из палитры в canvas

- Захватите узел в палитре левой кнопкой мыши.
- Перетащите в нужную точку canvas — узел появится с автогенерированным
  `id`. Можно перетаскивать после установки.
- Удаление: выделить узел, нажать **Delete** или **Backspace**.

### 3.3. Соединение узлов

- Из правой стороны узла-источника тяните стрелку к левой стороне
  узла-цели.
- Для **decision-узлов** автоматически создаются `ChoiceEdge` — это рёбра
  с привязкой к опции (поле `option_id`). На рёбрах хранятся:
  - `is_correct` — это «верный путь» или «дистрактор»;
  - `score_delta` — сколько баллов даёт переход;
  - `partial` — частичный кредит.
- Из `data` / `start` обычно один исходящий edge (линейное продвижение).

## 4. Узел `data`: HTML-контент и санитизация

В inspector доступно поле **Content HTML**. Используйте семантические
теги: `<h3>`, `<p>`, `<ul>`, `<table>`. Стили инлайн использовать
не нужно — применяются токены дизайн-системы.

### Правила безопасности

- HTML проходит **DOMPurify** на клиенте и backend-санитизер на сервере.
- **Изображения и вложения** допускаются только с префиксом `/media/`.
  Внешние URL (`https://...`, `http://...`) **запрещены** и будут
  вырезаны санитизером.
- Загружайте картинки через раздел *Media library*, затем используйте путь
  `/media/<file-uuid>.png`.
- `<script>`, `<iframe>`, `on*=` атрибуты автоматически удаляются.

### Пример HTML для эпидкейса

```html
<h3>Анамнез заболевания</h3>
<p>Курсант 3-го курса, 21 год. Жалобы: тошнота, рвота 4 раза, диарея 6 раз
за сутки. Заболел через 4 часа после обеда в столовой.</p>
<p><img src="/media/eda-stool-2026-03-14.png" alt="Анализ кала" /></p>
```

## 5. Узел `decision`: варианты и партиал-скоринг

Inspector предоставляет:

- **Allow multiple answers** (`allow_multiple`) — разрешить выбор нескольких
  вариантов одновременно.
- **Partial credit** (`partial_credit`) — частичные баллы за частично
  верный набор.
- Кнопка **«Add option»** — добавить вариант. Каждый вариант получает
  `id` (например, `option-1`) и `label` (текст для студента).

После добавления вариантов **обязательно** проведите рёбра от decision-узла
к следующим узлам. Для каждого ребра в инспекторе ребра укажите:

- `option_id` — какой вариант ведёт по этому пути;
- `is_correct` — да/нет (зелёный «Expected path» vs красный «Distractor»);
- `score_delta` — сколько баллов даёт этот выбор.

### Пример для эпидемиолога

> Вопрос: какое исследование заказать первым?
>
> - [ ] Бакпосев кала на патогенную флору (`is_correct=true`, `+10`)
> - [ ] Общий анализ крови (`is_correct=true, partial=true`, `+3`)
> - [ ] КТ брюшной полости (`is_correct=false`, `0`)

## 6. Узел `form`: поля, валидация и скоринг

Узел `form` ссылается на **form template** (`form_template_id`). Шаблон
определяет поля; их редактирование — через раздел *Form templates*.

Для каждого поля inspector позволяет задать:

- **`required`** — поле обязательно;
- **`validation_regex`** — например, `^[A-Z][0-9]{2}(\.\d{1,2})?$` для
  МКБ-10;
- **`score_value`** — балл за корректно заполненное поле;
- **`correct_value`** — хранится **только на сервере**, никогда не
  отдаётся клиенту в production. В режиме *Preview* (см. §10) виден
  в панели Insights.

Скоринг формы = сумма `score_value` по верно заполненным полям.

---

## 7. Узел `text_input`: ключевые слова и синонимы

Inspector:

- **Keywords** (через запятую): минимально необходимые слова в ответе.
  Например: `сальмонеллёз, групповая, столовая`.
- **Synonyms** — JSON-объект, например:
  ```json
  {
    "сальмонеллёз": ["salmonella", "сальмонеллез"],
    "столовая": ["пищеблок", "буфет"]
  }
  ```
- **Minimum length** (`min_length`) — минимальная длина ответа в символах.
- **Partial credit** — балл выдаётся пропорционально доле совпавших
  ключевых слов (если включён `partial`).

Backend нормализует ответ (lowercase, удаляет пунктуацию) и сверяет
с `keywords ∪ synonyms`.

---

## 8. Узел `final`: порог прохождения

Inspector:

- **Result type**: `correct` / `partial` / `incorrect` — что показать
  студенту на финальном экране.
- **Pass threshold** — процент `score / max_score`, начиная с которого
  попытка считается **passed**. Стандарт ВМедА: **70 %**.

Узлов `final` может быть несколько — каждый ведёт к собственному
сообщению (хорошо/плохо/средне).

---

## 9. Авто-сохранение и защита от потери данных

- Граф сохраняется автоматически **каждые 30 секунд** при наличии
  изменений (debounce, не на каждый drag).
- В заголовке редактора виден индикатор:
  - `● Saving...` — идёт запись;
  - `✓ Saved 14:32` — последнее успешное сохранение;
  - красный текст — ошибка автосейва (часто из-за валидации).
- Кнопка **Save** — ручное немедленное сохранение.
- При попытке закрыть вкладку с несохранёнными изменениями браузер
  покажет `beforeunload`-предупреждение.

> Если автосейв падает с 422 — откройте Inspector и проверьте, что
> у всех узлов заполнены обязательные поля.

## 10. Preview: режим студента + Insights

Кнопка **Preview** в шапке редактора открывает `/teacher/scenarios/:id/preview`.

- Слева: точный плеер кейса (тот же `CasePlayer`, что у студента),
  но **без записи попытки на backend**. Ответы существуют только локально.
- Справа: панель **Insights** (только для преподавателя):
  - список рёбер с пометкой `Expected path` / `Distractor`;
  - корректные значения полей форм;
  - keywords из text-input узлов.

Используйте preview, чтобы:

1. Пройти кейс «как студент» и понять UX.
2. Убедиться, что верный путь действительно ведёт к финалу.
3. Проверить читаемость HTML в data-узлах.

## 11. Publish: валидация графа

Перед публикацией backend выполняет **`graph_engine.validate_graph`**:

| Проверка                  | Ошибка / поведение                                     |
|---------------------------|--------------------------------------------------------|
| Ровно один `start`        | Иначе 422 «Scenario must have exactly one start node». |
| Хотя бы один `final`      | Иначе 422 «Scenario must have at least one final».     |
| Нет циклов                | DFS обнаруживает back-edges → 422.                     |
| Нет недостижимых узлов    | BFS от `start` → unreachable → 422.                    |
| `max_score > 0`           | Сумма всех `score_delta` + `score_value`.              |
| Decision: все опции имеют исходящие рёбра | Иначе 422.                              |

Если валидация прошла, статус переходит `draft → published`, кнопка
«Publish» становится «Unpublish».

## 12. Назначение сценария группе

1. Перейдите в **«Группы»** (`/teacher/groups`).
2. Выберите группу (например, `3-й курс, 5-й факультет`).
3. В блоке *Assigned scenarios* нажмите **«Assign»**.
4. Выберите опубликованный сценарий, укажите:
   - `available_from` — дата открытия;
   - `available_to` — дедлайн;
   - `max_attempts` — обычно `3`;
   - `is_visible` — показывать ли студентам сейчас.
5. После сохранения студенты увидят кейс в разделе «Мои кейсы».

## 13. Аналитика по сценарию

Откройте `/teacher/scenarios/:id/analytics`. Сверху — 4 KPI-плитки:

- **Прошли** (`completed / total_students`);
- **Средний балл** (%);
- **Среднее время** (мм:сс);
- **Правильный путь** — сколько раз пройден.

Ниже — 4 вкладки:

| Вкладка             | Что показывает                                                 |
|---------------------|----------------------------------------------------------------|
| **Тепловая карта**  | Граф, узлы окрашены по `avg_score_pct` (зелёный/жёлтый/красный). Кликаем — модал с детализацией. |
| **Распределение**   | Гистограмма распределения итоговых баллов по диапазонам 0–100. |
| **Рейтинг**         | Таблица студентов: ФИО, балл, время, путь.                     |
| **Слабые узлы**     | Узлы со средним < 50 %, отсортированы по числу посещений + типичная неверная реплика. |

### Экспорт

- Кнопка **«Скачать XLSX»** — экспорт ranking + score distribution.
- Кнопка **«Скачать PDF»** — отчёт со всеми KPI + heatmap (для печати,
  подшивки в учебно-методический комплекс).

Фильтр **Группа** позволяет смотреть метрики по конкретной группе или
по всем сразу.

## 14. Что НЕ делать

- **НЕ редактируйте опубликованный сценарий «вживую».** Если правки
  обязательны:
  1. Снимите с публикации (`Unpublish`) — статус вернётся в `draft`.
  2. Внесите изменения.
  3. Заново опубликуйте.
  Без этого студенты, проходящие кейс прямо сейчас, увидят рассинхрон
  и попытка будет помечена `abandoned`.
- **НЕ удаляйте узлы**, на которые ссылаются завершённые попытки —
  это ломает воспроизведение результатов. Используйте `Archive`.
- **НЕ храните корректные ответы в `data`-узлах**. Они санитизируются,
  но логически ответ принадлежит decision/form/text_input.
- **НЕ выставляйте `time_limit_sec` меньше 600** (10 мин) — студенты не
  успевают прочитать материал.
- **НЕ загружайте картинки больше 5 МБ** — медленные сети ВМедА.
